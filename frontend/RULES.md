# RULES

아래 규칙을 기본으로 따릅니다.

---

## 1. 브랜치 전략

- 기본 브랜치는 `main`을 사용합니다.
- 작업 브랜치는 기능/목적 단위로 생성합니다.
- 하나의 브랜치에는 하나의 목적(기능/버그)만 포함합니다.

### 브랜치 예시

```bash
feat/login-page
fix/header-style
refactor/api-client
docs/pr-template
```

---

## 2. 커밋 규칙

- 커밋 메시지는 반드시 한국어로 작성합니다.
- 반드시 `type(scope): subject` 형식을 사용합니다.
- subject는 짧고 명확하게 작성합니다.
- scope는 변경된 기능 또는 폴더 기준으로 작성합니다.

### 커밋 예시

```bash
feat(button): 버튼 UI 추가
fix(tooltip): 툴팁 위치 오류 수정
refactor(api): ky 인스턴스 구조 개선
docs(readme): PR 작성 규칙 추가
chore(page): 미사용 import 제거
```

---

## 3. PR 규칙

- PR 제목은 반드시 아래 형식을 사용합니다.

```bash
[브랜치명] - 변경 사항 요약
```

### PR 제목 예시

```bash
[feat/button-ui] - 버튼 UI 생성
[fix/tooltip-position] - 툴팁 위치 오류 수정
```

---

## 4. PR 설명 규칙

PR 설명에는 반드시 아래 내용을 포함합니다.

- 요약
- 주요 변경 파일
- 변경(추가) 이유

### PR 설명 템플릿

```md
## 요약

1. 변경 사항 요약

---

## 주요 변경 파일

| 파일        | 설명      |
| ----------- | --------- |
| `파일 경로` | 변경 내용 |

---

## 변경(추가) 이유

- 변경한 이유
```

### PR 설명 예시

```md
## 요약

1. PR 제목 컨벤션 추가
2. PR 설명 구조 통일
3. 주요 변경 파일 / 변경 이유 작성 규칙 추가

---

## 주요 변경 파일

| 파일                               | 설명                   |
| ---------------------------------- | ---------------------- |
| `.github/pull_request_template.md` | PR 템플릿 구조 정의    |
| `README.md`                        | PR 작성 규칙 문서화    |
| `docs/convention/pr-guide.md`      | 제목 및 설명 예시 추가 |

---

## 변경(추가) 이유

- PR 형식을 통일해 리뷰 가독성을 높이기 위함
- 변경 목적과 영향 범위를 빠르게 파악할 수 있도록 하기 위함
- 협업 시 리뷰어가 핵심 변경 사항을 쉽게 이해할 수 있도록 하기 위함
```

---

## 5. 코드 리뷰 규칙

- 머지 전 AI 코드 리뷰를 포함해 검토합니다.
- 리뷰 코멘트에 대한 처리(반영/보류 사유)를 명확하게 남깁니다.
- 범위가 큰 PR보다 작은 PR을 자주 올리는 것을 권장합니다.
- UI 변경 시 스크린샷 또는 짧은 영상 첨부를 권장합니다.

---

## 6. 머지 규칙

- 기본적으로 Squash Merge를 사용합니다.
- CI 또는 검증 실패 상태에서는 머지하지 않습니다.

---

## 7. 최소 체크리스트

- `pnpm lint` 통과
- 주요 경로 수동 확인
  - 렌더링
  - 네비게이션
  - 오류 상태
- 불필요한 console 제거
- 사용하지 않는 import 제거
