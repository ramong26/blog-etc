# CONTRIBUTING

협업 시 아래 규칙을 기본으로 따릅니다.

## 1. 브랜치 전략

- 기본 브랜치: `main`
- 작업 브랜치 예시: `feat/login-page`, `fix/header-style`
- 하나의 브랜치에는 하나의 목적(기능/버그)을 담습니다.

## 2. 커밋 규칙

- 커밋 메시지는 변경 의도가 드러나게 작성합니다.
- 권장 예시:
	- `feat: add post list skeleton`
	- `fix: handle null author profile`
	- `refactor: split api client module`

## 3. PR 규칙

- PR에는 변경 배경, 핵심 변경점, 확인 방법을 포함합니다.
- UI 변경이 있으면 스크린샷 또는 짧은 영상 첨부를 권장합니다.
- 범위가 큰 PR보다는 작은 PR을 자주 올립니다.

## 4. 코드 리뷰 규칙

- 머지 전 AI 코드리뷰를 포함해 검토합니다.
- 리뷰 코멘트에 대한 처리(반영/보류 사유)를 명확히 남깁니다.

## 5. 머지 규칙

- 머지는 스쿼시 머지를 기본으로 사용합니다.
- CI/검증이 실패한 상태에서는 머지하지 않습니다.

## 6. 최소 체크리스트

- `pnpm lint` 통과
- 주요 경로 수동 확인(렌더링/네비게이션/오류 상태)
- 불필요한 콘솔 로그 제거
