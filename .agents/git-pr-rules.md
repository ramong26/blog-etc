# Git / PR 규칙

## 브랜치 규칙

- 기본 브랜치는 `main` 사용함
- 작업 브랜치는 기능 또는 목적 단위로 생성함
- 하나의 브랜치에는 하나의 목적만 포함함

### 브랜치 예시

```bash
feat/login-page
fix/header-style
refactor/api-client
docs/pr-template
```

---

## 커밋 메시지 규칙

- 커밋 메시지는 반드시 한국어로 작성함
- 반드시 `type(scope): subject` 형식 사용함
- scope는 변경된 기능 또는 폴더 기준으로 작성함
- subject는 짧고 명확하게 작성함

### 커밋 예시

```bash
feat(button): 버튼 UI 추가
fix(tooltip): 툴팁 위치 오류 수정
refactor(api): ky 인스턴스 구조 개선
docs(readme): PR 작성 규칙 추가
chore(page): 미사용 import 제거
```

---

## PR 제목 규칙

- PR 제목은 반드시 아래 형식 사용함

```bash
[브랜치명] - 변경 사항 요약
```

### 제목 예시

```bash
[feat/button-ui] - 버튼 UI 생성
[fix/tooltip-position] - 툴팁 위치 오류 수정
```

---

## PR 설명 규칙

PR 설명에는 반드시 아래 내용 포함함

- 요약
- 주요 변경 파일
- 변경 추가 이유

### PR 설명 템플릿

```md
## 요약

1. 변경 사항 요약

---

## 주요 변경 파일

| 파일        | 설명      |
| ----------- | --------- |
| `파일 경로` | 변경 내용 |

---

## 변경 추가 이유

- 변경한 이유
```

### PR 설명 예시

```md
## 요약

1. PR 제목 컨벤션 추가
2. PR 설명 구조 통일
3. 주요 변경 파일과 변경 이유 작성 규칙 추가

---

## 주요 변경 파일

| 파일                    | 설명                   |
| ----------------------- | ---------------------- |
| `AGENTS.md`             | 작업 진입 규칙 정리    |
| `frontend/app/page.tsx` | 미사용 import 제거     |
| `backend/README.md`     | 프로젝트 문서 업데이트 |

---

## 변경 추가 이유

- PR 형식을 통일해 리뷰 가독성을 높이기 위함
- 변경 목적과 영향 범위를 빠르게 파악하기 위함
- 리뷰어가 핵심 변경 사항을 쉽게 이해하기 위함
```
